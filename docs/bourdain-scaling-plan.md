# Bourdain Pipeline Scaling Plan: kjtcom Project

This document outlines the scaling plan for processing the remaining Bourdain videos through the kjtcom pipeline, targeting completion within 2-3 iterations.

## 1. Pipeline Analysis Summary

Reviewing previous projects provides valuable insights into expected performance and potential bottlenecks:

*   **CalGold (390 videos, 899 entities)**: Entity yield ~2.3 entities/video.
*   **RickSteves (1865 videos, 4182 entities)**: Entity yield ~2.2 entities/video.
*   **TripleDB (805 videos, 1100 entities)**: Entity yield ~1.4 entities/video.
*   **Bourdain Phase 1 (30 videos, 96 entities)**: Entity yield ~3.2 entities/video.

**Key Observations:**

*   **Higher Bourdain Yield**: Bourdain content consistently yields a higher number of entities per video (~3.2) compared to previous projects. This is likely due to Bourdain's focus on specific locations, food, and people, making the content richer in identifiable entities. We will use this higher yield for projections.
*   **GPU Bottleneck**: Transcription using `faster-whisper` on the RTX 2080 SUPER is the primary time-consuming phase, averaging ~10 minutes per video.
*   **API Constraints**: Nominatim's rate limit (1 req/sec) and Google Places' daily free quota require careful management.
*   **Free Tier Reliance**: Gemini Flash extraction relies on the free tier, which is generally robust for this scale.

## 2. Entity Yield Projections for Bourdain

Based on the Bourdain Phase 1 yield of ~3.2 entities/video:

*   **Remaining Videos**: 84 videos (from video 31 to 114).
*   **Projected Entities**: 84 videos * 3.2 entities/video = 268.8 entities.
*   **Rounded Projection**: Approximately **269 new entities**.

## 3. Concrete Phase-by-Phase Execution Plan

The pipeline consists of 7 sequential phases. Processing will occur in graduated batches of 10 videos (or fewer for the final batch).

### Phase 1: Acquire (yt-dlp)

*   **Purpose**: Download YouTube videos.
*   **Tool**: `yt-dlp`
*   **Input**: List of YouTube URLs/IDs for the current batch.
*   **Output**: MP4 video files.
*   **Command Example**:
    ```bash
    yt-dlp -f 'bestvideo[ext=mp4]+bestaudio[ext=m4a]/best[ext=mp4]' \
           --output "%(id)s.%(ext)s" \
           --batch-file <path_to_video_urls.txt>
    ```
*   **Notes**: This phase is I/O bound. It can be run for an entire iteration's videos at once, or per batch for consistency.

### Phase 2: Transcribe (faster-whisper CUDA)

*   **Purpose**: Generate accurate transcripts from video audio.
*   **Tool**: `faster-whisper` (CUDA)
*   **Input**: MP4 video files from Phase 1.
*   **Output**: SRT/VTT/TXT transcript files.
*   **Constraint**: RTX 2080 SUPER 8GB VRAM.
*   **Performance**: ~10 minutes per video.
*   **Batching**: **Strictly 10 videos per batch, processed sequentially.**
*   **Command Example (assuming a wrapper script `transcribe_batch.py`):**
    ```bash
    # Example: Process videos for batch 1 (video_id_31.mp4 to video_id_40.mp4)
    python transcribe_batch.py --model large-v2 --device cuda \
                               --output_format srt \
                               --video_files video_id_31.mp4 video_id_32.mp4 ... video_id_40.mp4
    ```
*   **Notes**: This is the GPU bottleneck. Monitor GPU usage and temperature.

### Phase 3: Extract (Gemini Flash)

*   **Purpose**: Identify and extract entities (e.g., locations, people, dishes) from transcripts.
*   **Tool**: Gemini Flash API.
*   **Input**: Transcript files from Phase 2.
*   **Output**: Raw JSON entities.
*   **Constraint**: Free tier usage.
*   **Notes**: Ensure robust prompt engineering for consistent extraction. Implement retry logic for API calls.

### Phase 4: Normalize (schema v3)

*   **Purpose**: Standardize extracted entities to conform to `schema v3`.
*   **Tool**: Custom Python script.
*   **Input**: Raw JSON entities from Phase 3.
*   **Output**: Normalized JSON entities.
*   **Notes**: This is a CPU-bound process and typically very fast. Regular review of normalized data is crucial for quality.

### Phase 5: Geocode (Nominatim)

*   **Purpose**: Convert location names into geographical coordinates (latitude, longitude).
*   **Tool**: Nominatim API.
*   **Input**: Normalized entities with location names from Phase 4.
*   **Output**: Geocoded entities.
*   **Constraint**: **Nominatim rate limit: 1 request/second.**
*   **Mitigation**: Implement a mandatory delay between API calls.
*   **Command/Logic Example**:
    ```python
    import requests
    import time

    def geocode_entity(location_name):
        url = f"https://nominatim.openstreetmap.org/search?q={location_name}&format=json&limit=1"
        headers = {'User-Agent': 'kjtcom-bourdain-project/1.0'} # Required User-Agent
        response = requests.get(url, headers=headers)
        response.raise_for_status()
        data = response.json()
        time.sleep(1.1) # Enforce 1.1 second delay
        return data[0] if data else None

    # Iterate through entities and call geocode_entity
    ```
*   **Notes**: For a batch of 10 videos yielding ~32 entities, this phase will take at least ~35 seconds.

### Phase 6: Enrich (Google Places)

*   **Purpose**: Add supplementary information (e.g., place types, addresses, ratings) to geocoded entities.
*   **Tool**: Google Places API.
*   **Input**: Geocoded entities from Phase 5.
*   **Output**: Enriched entities.
*   **Constraint**: Daily free quota.
*   **Mitigation**: Monitor API usage closely via the Google Cloud Console. Implement caching for frequently queried places to minimize API calls. Prioritize essential enrichment fields.

### Phase 7: Load (staging)

*   **Purpose**: Store the final processed entities into the staging environment.
*   **Tool**: Custom script (e.g., SQL inserts, JSON file writes).
*   **Input**: Enriched entities from Phase 6.
*   **Output**: Data in staging database/files.
*   **Notes**: This phase should be robust with error handling and logging.
